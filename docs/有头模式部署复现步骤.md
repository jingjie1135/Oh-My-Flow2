# Flow2API 有头模式部署复现步骤

## 文档目的

本文档用于固化 Flow2API 在 Docker 环境下部署有头浏览器验证码模式的标准复现流程。

本文仅覆盖当前已确认可成功完成有头浏览器打码与生图链路验证的 `browser` 模式，不包含 `personal` 模式的部署结论。

文档内容覆盖以下方面：

- 目录准备
- 配置校验
- 镜像构建
- 服务启动
- 运行验证
- 关键实现点说明
- 常见故障排查

---

## 适用范围

本文档对应当前仓库中的以下文件组合：

- `Dockerfile.headed`
- `docker/entrypoint.headed.sh`
- `docker-compose.browser.yml`
- `config/setting.browser.toml`

## 前置条件

执行前应确认以下条件已满足：

1. 已安装 Docker
2. 已安装 Docker Compose v2
3. 已获取当前项目源码
4. 具备修改 `config/*.toml` 配置文件的权限

项目目录示例：

```text
<project-root>/
```

其中 `<project-root>` 表示当前仓库根目录，不应替换为仅适用于单一机器环境的固定绝对路径。

## 部署目标

本流程的目标是完成以下服务的部署与验证：

- `browser` 模式：按请求启动浏览器处理验证码，并作为当前已确认可成功完成有头生图链路验证的模式

部署完成后，默认访问端口如下：

- `browser`：`38101`

## 步骤一：进入项目目录

```powershell
cd <project-root>
```

建议在仓库根目录执行全部命令，以确保以下路径引用成立：

- `Dockerfile.headed`
- `docker-compose.browser.yml`
- `config/setting.browser.toml`
- `data/browser`
- `tmp/browser`

---

## 步骤二：准备模式专用配置文件

有头模式不建议共用默认 `config/setting.toml`，应为 `browser` 模式准备独立配置文件。

若目标文件尚未创建，可先基于示例文件复制：

```powershell
copy config\setting_example.toml config\setting.browser.toml
```

配置完成后，应至少确认以下字段：

### `config/setting.browser.toml`

```toml
captcha_method = "browser"
```

如存在上游认证、代理或 token 相关参数，也应按实际部署环境补全。

---

## 步骤三：创建独立数据目录与临时目录

为避免 SQLite 运行状态、缓存文件以及模式配置相互污染，应为 `browser` 模式准备独立的数据目录与临时目录。

建议执行以下命令：

```powershell
New-Item -ItemType Directory -Force -Path "data","tmp","data/browser","tmp/browser"
```

目录结构如下：

```text
data/
└── browser/

tmp/
└── browser/
```

## 步骤四：校验 Compose 配置

在启动容器前，应先验证 Compose 文件可被正确解析。

### 校验 `browser` 模式

```powershell
docker compose -f docker-compose.browser.yml config
```

若上述命令能够正常输出展开后的配置，则说明以下内容基本无误：

- 构建上下文路径
- 挂载路径
- 环境变量格式
- 端口映射
- 配置文件引用

---

## 步骤五：启动 `browser` 有头模式

执行以下命令构建并启动 `browser` 模式：

```powershell
docker compose -f docker-compose.browser.yml up -d --build
```

该命令会完成以下动作：

1. 使用 `Dockerfile.headed` 构建镜像
2. 安装 Playwright 及 Chromium 运行依赖
3. 挂载 `config/setting.browser.toml`
4. 挂载独立的 `data/browser` 与 `tmp/browser`
5. 启动容器并在容器内部初始化有头显示环境

---

## 步骤六：检查容器状态

服务启动后，应首先确认容器是否处于 `Up` 状态。

### 检查 `browser` 模式

```powershell
docker compose -f docker-compose.browser.yml ps
```

若服务状态为 `Up`，说明容器已成功完成基础启动。

---

## 步骤七：检查 `browser` 模式日志

仅确认容器启动并不足以说明有头浏览器环境已经可用，还应检查日志中是否存在虚拟显示与浏览器预热成功信息。

执行命令：

```powershell
docker logs flow2api-browser --tail 120
```

预期关键日志示例：

```text
[entrypoint] starting virtual display :99 (1280x720x24)
[entrypoint] starting fluxbox window manager
[entrypoint] starting flow2api (headed browser mode)
[BrowserCaptcha] Token-0 shared browser started
[BrowserCaptcha] warmed browser slot 0
```

若日志中出现以下报错，则说明有头显示环境未正确初始化：

```text
Missing X server or $DISPLAY
```

---

## 步骤八：验证 HTTP 入口

在容器状态和日志均正常后，应进一步验证服务入口是否可访问。

### 验证首页

```powershell
Invoke-WebRequest -UseBasicParsing -Uri "http://127.0.0.1:38101/" -TimeoutSec 10
```

### 验证登录页

```powershell
Invoke-WebRequest -UseBasicParsing -Uri "http://127.0.0.1:38101/login" -TimeoutSec 10
```

预期结果：返回状态码 `200`。

## 有头模式成功运行的关键实现点

### 1. `Dockerfile.headed` 提供图形与浏览器依赖

有头模式依赖以下关键包：

- `xvfb`
- `fluxbox`
- `xauth`

这些依赖用于在容器内部构建虚拟显示环境，使 Playwright/Chromium 能够以接近真实桌面浏览器的方式运行。

### 2. `entrypoint.headed.sh` 负责启动虚拟显示环境

容器启动时，脚本会执行以下核心流程：

1. 设置 `DISPLAY`，默认值为 `:99`
2. 启动 `Xvfb`
3. 启动 `fluxbox`
4. 探测 Chromium 可执行文件路径
5. 启动 `python main.py`

该流程是“有头模式可用”的核心前提。

### 3. `docker-compose.browser.yml` 提供必要运行时参数

`docker-compose.browser.yml` 中包含以下关键参数：

- `dockerfile: Dockerfile.headed`
- `ALLOW_DOCKER_HEADED_CAPTCHA=true`
- `shm_size: "2gb"`

其中：

- `ALLOW_DOCKER_HEADED_CAPTCHA=true` 用于显式启用 Docker 内有头验证码逻辑
- `shm_size: "2gb"` 用于为 Chromium 提供足够的共享内存，降低崩溃概率

## 命令清单（按部署方式分类）

为便于在不同环境中快速复现，以下命令按部署方式分别整理。

### 方式一：Docker 有头模式部署（推荐）

适用场景：

- 需要最接近当前已验证成功路径的复现方式
- 需要直接使用 `Dockerfile.headed` 提供的 `Xvfb + Fluxbox + Chromium` 环境
- 需要与当前文档中的验证步骤保持一致

```powershell
cd <project-root>

New-Item -ItemType Directory -Force -Path "data","tmp","data/browser","tmp/browser"

docker compose -f docker-compose.browser.yml config

docker compose -f docker-compose.browser.yml up -d --build

docker compose -f docker-compose.browser.yml ps

docker logs flow2api-browser --tail 120

Invoke-WebRequest -UseBasicParsing -Uri "http://127.0.0.1:38101/" -TimeoutSec 10
Invoke-WebRequest -UseBasicParsing -Uri "http://127.0.0.1:38101/login" -TimeoutSec 10
```

### 方式二：Linux 非 Docker 本地部署

适用场景：

- 需要直接在 Linux 主机上运行项目
- 需要避开 Docker 挂载与容器网络因素
- 需要在宿主机上自行提供浏览器依赖与图形环境

说明：

- 仓库已提供 `python main.py` 本地启动入口
- `requirements.txt` 中已包含 `playwright` 与 `nodriver`
- 若要复现有头浏览器相关能力，Linux 主机仍需具备浏览器运行所需的系统依赖与可用显示环境（物理桌面、远程桌面或虚拟显示环境）

#### 方案 A：Linux 图形桌面环境

```bash
cd <project-root>

python3 -m venv .venv
source .venv/bin/activate

pip install -r requirements.txt

python -m playwright install --with-deps chromium

mkdir -p data/browser tmp/browser

cp config/setting_example.toml config/setting.browser.toml
# 编辑 config/setting.browser.toml，确认 captcha_method="browser"

python main.py
```

启动后，可使用以下命令验证：

```bash
curl -I http://127.0.0.1:8000/
curl -I http://127.0.0.1:8000/login
```

#### 方案 B：Linux 无图形桌面服务器（使用 Xvfb）

适用场景：

- 服务器没有物理桌面或远程桌面环境
- 需要通过虚拟显示方式运行 browser 有头模式

```bash
cd <project-root>

python3 -m venv .venv
source .venv/bin/activate

pip install -r requirements.txt

python -m playwright install --with-deps chromium

mkdir -p data/browser tmp/browser

cp config/setting_example.toml config/setting.browser.toml
# 编辑 config/setting.browser.toml，确认 captcha_method="browser"

sudo apt update
sudo apt install -y xvfb fluxbox xauth

Xvfb :99 -screen 0 1280x720x24 &
export DISPLAY=:99
fluxbox >/tmp/flow2api-fluxbox.log 2>&1 &

python main.py
```

启动后，可使用以下命令验证：

```bash
curl -I http://127.0.0.1:8000/
curl -I http://127.0.0.1:8000/login
```

## 常见故障与排查建议

### 问题一：容器已启动，但日志中出现 `Missing X server or $DISPLAY`

排查方向：

- 检查是否使用了 `Dockerfile.headed`
- 检查 `entrypoint.headed.sh` 是否实际启动了 `Xvfb`
- 检查容器是否正确设置 `DISPLAY`

### 问题二：浏览器频繁崩溃或页面无法稳定加载

排查方向：

- 检查 Compose 中是否设置了 `shm_size: "2gb"`
- 检查宿主机是否存在严重资源不足

### 问题三：容器可访问，但验证码相关能力不可用

排查方向：

- 检查配置文件中的 `captcha_method` 是否为 `browser`
- 检查上游 token、代理或认证配置是否齐全
- 检查日志中浏览器 slot 是否完成预热

## 结论

Flow2API 当前已确认可成功复现的有头生图链路，依赖以下运行时组合：
